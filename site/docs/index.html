Start / Overview

Asylum documentation

Asylum is a native Agent Development Environment. It runs coding agents in isolated Git worktrees, records each attempt, and gives you one place to compare, verify, and merge the result.

Built features in this manual are backed by a working UI, CLI, or tested core.

Core model

Project

A Git repository registered with Asylum.

Task

The outcome you want agents to complete.

Run

One agent attempt in one branch and worktree.

Workflow

  1. Set upOpen a repository and verify Git, branches, agents, and project tooling.
  2. DispatchWrite one task and select one or more installed agents.
  3. ObserveWatch each PTY, inspect its worktree, and retain terminal history.
  4. ReviewCompare diffs, comments, and detected project checks.
  5. FinishMerge the selected run or open a pull request.

Start / Install

Install

Asylum currently builds from source. You need Rust, Git, and at least one supported agent CLI.

Prerequisites

  • Rust toolchain with Cargo
  • Git available on PATH
  • One signed-in coding-agent CLI
  • Bun for JavaScript project checks

Build and launch

Terminal
git clone https://github.com/wess/asylum.git
cd asylum
cargo run -p app

The development binary is named asylumdev, so it never collides with an installed release binary named asylum.

First build

Cargo fetches pinned GPUI, guise-ui, and libsinclair dependencies. This build takes longer than later launches.

Verify the workspace

Terminal
cargo test --workspace
cargo clippy --all-targets

Start / First task

Run your first task

Start with one repository and one agent. Add parallel runs after the complete workflow feels familiar.

  1. Open a repositorySelect an existing Git repository. For a plain folder, Asylum asks before creating a repository and empty initial commit.
  2. Read Setup doctorResolve blocked Git, branch, worktree, agent, Bun, or Cargo checks.
  3. Verify an agentRun the setup test. “Installed” means the executable exists; “verified” means it completed a real authenticated run.
  4. Describe the outcomeUse a task template or write a concrete result with acceptance criteria.
  5. Create and runAsylum creates the branch, worktree, durable run row, and PTY in the background.
  6. Review the resultOpen Review, read the diff and checks, then merge or continue the run.
Good first task

“Add a health endpoint returning JSON, include one test, and update the README.” It names a result and gives the agent a clear finish line.

Operate / Agents and fan-out

Agents and fan-out

Asylum resolves an agent definition plus your preferences into a spawn specification. The app creates isolation before it launches the process.

Built-in registry

The registry includes Claude Code, Codex, OpenCode, Gemini, Aider, Cursor, and additional CLI agents. Settings can override the executable and append arguments without changing the built-in catalog.

When to fan out

One agent

Small fixes, known implementation path, or low-cost work.

Several agents

Architecture choices, UI alternatives, uncertain bugs, or work where comparison improves confidence.

Isolation contract

Every plan receives a unique branch and absolute worktree path. The app persists that path before execution, so retries and review follow-ups continue in the same worktree.

Generated shape
task: Add cache invalidation
├── asylum/add-cache-42-claude-code  → .asylum/worktrees/...
├── asylum/add-cache-42-opencode     → .asylum/worktrees/...
└── asylum/add-cache-42-gemini       → .asylum/worktrees/...

Operate / Runs and terminals

Runs and terminals

A run moves through queued, running, succeeded, failed, or cancelled states. Its worktree and transcript survive after the process exits.

Run controls

Cancel
Stops a queued or running attempt without deleting its worktree.
Retry
Starts another attempt in the same worktree and increments the attempt count.
Follow-up
Sends context to a live stdin-capable agent or queues a continuation.
Full terminal
Opens the persistent terminal — powered by libsinclair — in a dedicated workspace tab.

Restart recovery

A PTY cannot survive the application process. On restart, Asylum marks interrupted runs failed and preserves their branches, worktrees, output, and retry path.

Operate / Fleet control

Fleet control

See which agent needs you, race a preset of agents in one pick, and let a running agent orchestrate the rest of the fleet.

Semantic states

Beyond queued/running/succeeded, each run shows a live activity classified from its output, so a fan-out board tells you at a glance which agent is waiting on you.

working
Actively thinking, editing, or running a command.
blocked
Stopped at a prompt, waiting for your answer — the state worth surfacing first.
done
Printed a completion marker; awaiting review.
idle
Initialised but producing no recognisable signal.

Fan-out layouts

A layout is a named preset in settings.json — race a set of agents in one gesture. Built-ins duel, triad, and swarm ship out of the box; define your own with an optional per-layout concurrency cap. Inspect them with asylum layout list.

The control surface

A running agent can orchestrate the fleet from inside its worktree over a local JSON API it learns from a skill (asylum control skill). The app injects ASYLUM_CONTROL_URL, ASYLUM_TASK_ID, and ASYLUM_RUN_ID into every run.

asylum control status                 # your run + siblings, with live activity
asylum control spawn codex "write the tests"
asylum control activity blocked       # report yourself before you ask a question
asylum wait run 42 --status succeeded # block until a sibling finishes

Write-effects (spawning a helper run, running checks) are queued and drained by the desktop app, so the API stays safe and pure. An append-only event log streams over both the control and companion servers, so a phone or an agent can follow the fleet without polling.

Operate / Review and merge

Review and merge

Select a run, then review its code, transcript, and checks together. Merge and pull-request actions stay blocked until the run is safe to finish.

Inline feedback

Click a diff line to attach a comment. “Send review to agent” batches unresolved comments into one continuation prompt and resolves them only after the queue operation succeeds.

Automatic checks

Asylum detects checks by project ecosystem. JavaScript projects use declared Bun scripts; Rust projects use Cargo. Each result is classified as pass, fail, or skipped.

Merge preflight

  • The selected run is finished.
  • Detected checks are not active or failing.
  • The base worktree has no uncommitted user changes.
  • The selected branch can merge without conflicts.

The final confirmation performs the merge. Cleanup removes clean finished worktrees and preserves dirty worktrees and branches.

Operate / Project memory

Project memory

Project knowledge remains ordinary Markdown. Store it privately in Asylum’s data directory or in the repository’s notes/ directory.

Wiki-style syntax

cacheinvestigation.md
---
title: Cache investigation
type: investigation
tags: [backend, reliability]
---

# Cache investigation

The decision is in [[Cache policy|the policy]]. #active

Properties, tags, wiki links, backlinks, autocomplete, and templates are indexed natively. Renaming a note updates incoming links across the vault.

Agent context

Create task
Turns the note into a task and makes it the source of truth.
Attach to run
Adds the note’s Markdown to an existing run’s durable context.
Send selection
Sends the exact selected passage to the current run.

Generated tasks, runs, checks, and pull requests append durable links back into the note.

Operate / Unified search

Unified search

One input searches repository files, notes, task prompts, run metadata, errors, and persisted terminal transcripts.

Sources

Files

Ripgrep with a Git grep fallback.

Notes

Title, path, body, tags, and linked content.

Tasks

Task titles and original prompts.

Runs

Agent, branch, output, and durable errors.

Selecting a result opens the owning surface and record, instead of leaving you in a detached search screen.

Extend / Project configuration

Project configuration

Commit asylum.toml at the repository root for shared defaults. Keep personal preferences in settings.json.

asylum.toml
base_branch = "main"
default_agents = ["opencode"]
setup = ["bun install"]

[env]
RUST_BACKTRACE = "1"

Setup commands run once in every new worktree before the agent starts. A failed command creates a durable failed run with the worktree path and error.

User settings

Open Settings or edit $XDG_CONFIG_HOME/asylum/settings.json. The file accepts JSON with comments and live-reloads theme and keybindings. Invalid values produce diagnostics while valid settings remain active. Alongside agents and the editor, it holds layouts (named fan-out presets), companion (the mobile server), and control (the agent control surface).

Extend / Plugins

Plugins

Plugins use a plugin.toml manifest and declare every capability they need. They can contribute commands, panels, webviews, triggers, and agent tools.

plugin.toml
id = "example.review"
name = "Review helper"
runtime = "process"
command = ["bun", "run", "index.ts"]
capabilities = ["notify"]

[[command]]
id = "summarize"
title = "Summarize selected run"

Runtime choices

Process

Newline JSON over stdio, one-shot or persistent session.

WASM

Sandboxed with wasmi; only declared host functions are linked.

Install and discover

A plugin is just a directory with a plugin.toml, so installing one is a shallow clone. Community plugins tag their repo with the asylum-plugin GitHub topic.

asylum plugin install acme/asylum-reviewr   # clone into the plugins dir
asylum plugin list                          # what's installed
asylum plugin search                        # community plugins, by topic

Extend / CLI

CLI

The asylum binary exposes worktree operations, headless runs, search, and computer-use commands.

Terminal
asylum worktree list /path/to/repo
asylum run opencode "Add a health endpoint"
asylum search /path/to/repo "cache key"
asylum control status                 # fleet control from inside a worktree
asylum wait run 42 --activity blocked
asylum plugin install acme/asylum-reviewr
asylum layout list
asylum snapshot
asylum click 420 280
asylum fill "project name"

The GUI and CLI share core crates for planning, Git operations, search, and execution. The CLI does not duplicate orchestration rules.

Extend / Troubleshooting

Troubleshooting

Agent is installed but not verified

Open Settings → Agents and run the setup test. Verification checks the executable, authentication, and real launch path.

A run failed after restarting Asylum

This is expected for a process that was active during shutdown. Retry the run to continue in its preserved worktree.

Merge is blocked

Resolve failed or active checks, clean user changes from the base worktree, and review any reported merge conflict.

A worktree will not delete

Asylum preserves dirty worktrees. Commit, move, or discard those changes explicitly before cleanup.

JavaScript checks are skipped

Declare typecheck, lint, or test scripts and make Bun available on PATH.

Settings contain a diagnostic

Fix the named key in settings.json. Other valid values remain active and defaults replace only invalid entries.

Collect a useful report

Include the task and run IDs, agent name, branch, worktree path, terminal output, failed check output, and the exact action that produced the problem.
