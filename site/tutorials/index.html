Beginner15 minutes

Your first isolated task

Complete the full Asylum loop with one agent and a low-risk repository task.

You will learnSetup doctorAgent verificationRun lifecycleReview

Before you start

Choose an existing Git repository with a clean base branch. Sign in to one supported agent CLI in your shell.

  1. 1

    Open the repository

    Launch Asylum, select Open repository, and choose the project root. Keep initialization off unless the folder is intentionally not a Git repository.

  2. 2

    Clear Setup doctor

    Read every row. Fix blocked Git, repository, base branch, worktree, agent, and toolchain checks before dispatching work.

  3. 3

    Verify one agent

    Open Agent settings and run the setup test. Continue only after the agent changes from installed to verified.

  4. 4

    Write a bounded task

    Use: “Add a health endpoint returning JSON, include one test, and document the route.” Select only the verified agent.

  5. 5

    Create and run

    Watch the run move from queued to running. Open its terminal if you need the complete live interaction.

  6. 6

    Review the evidence

    Open Review. Read the changed files, terminal output, and detected checks. Do not merge unless the code and checks match the requested outcome.

Checkpoint

You should have one finished run with a preserved branch, worktree, transcript, diff, and check result.

Beginner25 minutes

Compare two agent attempts

Use fan-out when two independent implementations can expose tradeoffs.

You will learnAgent selectionParallel runsDiff comparisonWinner selection
  1. 1

    Choose a comparison task

    Pick work with more than one valid implementation, such as “Add request caching with tests and document invalidation behavior.”

  2. 2

    Select two verified agents

    Prefer agents with different strengths. Asylum will create one branch and worktree for each.

  3. 3

    Dispatch once

    Create the task and launch both runs. Do not edit either generated worktree while agents are active.

  4. 4

    Read outcomes in the same order

    For each run, compare scope, changed files, tests, terminal reasoning, and check status. Use the same criteria for both.

  5. 5

    Select, then preflight

    Select the stronger run. Asylum checks failed verification, dirty base work, and merge conflicts before enabling the final action.

Decision rule

Choose the implementation that best satisfies the task with the lowest maintenance cost, not the run with the most changed files.

Beginner20 minutes

Build project memory

Create knowledge that humans and agents can both use without leaving Markdown.

You will learnVault modesTemplatesWiki linksTask context
  1. 1

    Choose storage

    Use Private for personal notes. Switch to Repository when the team should review and version notes under notes/.

  2. 2

    Create an investigation

    Use the Investigation template. Add YAML properties for owner and status, then add inline tags such as #active.

  3. 3

    Link a decision

    Create a Decision note. In the investigation, type [[ and select the decision from autocomplete. Confirm both outgoing links and backlinks.

  4. 4

    Create a task from the note

    Select the investigation and choose Create task. The note becomes durable task context and receives a task link.

  5. 5

    Inspect automatic references

    Run the task. The generated run and later checks or pull request append links back into the Markdown.

Checkpoint

Search for a phrase from the note and confirm results include the note, generated task, and attached run.

Intermediate30 minutes

Run a review loop

Convert exact code concerns into one structured continuation in the same worktree.

You will learnLine annotationsBatch feedbackSame-worktree retryResolution
  1. 1

    Select a finished run

    Open Review and verify you are looking at the intended branch and worktree.

  2. 2

    Annotate exact lines

    Click a changed line and write an actionable comment: name the problem, expected behavior, and evidence needed.

  3. 3

    Review the comment batch

    Add all related comments before sending. Keep unrelated changes in a separate continuation.

  4. 4

    Send review to agent

    Asylum creates another attempt in the same worktree. Comments resolve only after the continuation is queued successfully.

  5. 5

    Re-run verification

    Inspect the new diff and check results. Confirm the fix did not expand scope or regress previous behavior.

Strong review comment

“This error path drops the original cause. Preserve it in the returned message and add a test for a missing executable.”

Intermediate25 minutes

Make worktrees reproducible

Commit the project defaults every agent needs before work begins.

You will learnasylum.tomlSetup commandsEnvironmentDetected checks
  1. 1

    Add shared defaults

    Create asylum.toml with the real base branch and a conservative default agent list.

  2. 2

    Declare setup

    Add deterministic commands such as bun install --frozen-lockfile. Setup runs once in each new worktree.

  3. 3

    Expose required environment

    Add non-secret environment defaults. Keep credentials in the user environment, not project configuration.

  4. 4

    Declare package checks

    Add explicit typecheck, lint, and test scripts. Asylum does not guess missing JavaScript tooling.

  5. 5

    Test a fresh worktree

    Run a small task and confirm setup completes before the agent launches and all expected checks appear after success.

Checkpoint

A new run should need no manual dependency installation inside its generated worktree.

Intermediate25 minutes

Send visual feedback

Turn a browser observation into element-level implementation context.

You will learnBrowser surfaceDesign modePinsAgent feedback
  1. 1

    Open the local page

    Start the project’s development server and open its URL in Asylum’s Browser surface.

  2. 2

    Enable Design mode

    Toggle Design mode, then click the exact element that needs work. Asylum captures its selector, HTML, and computed CSS.

  3. 3

    Write outcome-focused feedback

    Describe the desired result, not a guessed CSS implementation. Save the note as a numbered pin.

  4. 4

    Capture related elements

    Add pins only when they belong to the same change. Keep unrelated visual work in another batch.

  5. 5

    Send the batch

    Choose the owning run and send all pins as one structured prompt. Verify the result in the same browser surface.

Useful note

“On narrow screens this action row clips the final button. Keep all actions visible without horizontal scrolling.”

Advanced30 minutes

Connect tools through the MCP gateway

Configure each service once, then expose a scoped and auditable tool catalog to every managed agent.

You will learnNamespacesKeep secretsTool filteringExposure modes

Before you start

Install one local MCP server or obtain a Streamable HTTP MCP endpoint. Unlock Keep before starting the app when the upstream needs a credential.

  1. 1

    Enable the loopback gateway

    In Settings → MCP gateway, enable the service and keep its bind address on 127.0.0.1. The app refuses non-loopback binds.

  2. 2

    Configure an upstream

    Edit mcp_servers in settings.json. Use command and args for stdio, or transport: "http" and url for Streamable HTTP. Give it a lowercase namespace such as github.

  3. 3

    Resolve credentials through Keep

    Run asylum keep set githubpat. Reference it as {secret:githubpat} in a stdio environment value, or set secret: "githubpat" for HTTP authorization. The agent never receives the secret.

  4. 4

    Reduce the exposed surface

    Use allow and deny to publish only required tools. Set project when a server belongs to one repository; a project-scoped server shadows a global server with the same name.

  5. 5

    Choose an exposure mode

    Use direct for a small catalog. Use search for a broad fleet: agents receive only asylum_find_tool and asylum_call_tool, then load definitions when needed.

  6. 6

    Verify from a managed run

    Launch a run and execute asylum mcp list. Confirm the expected namespaced tools appear, such as github__list_issues. Use asylum mcp stdio when an agent client supports MCP only over stdio.

Checkpoint

The run sees only the intended namespaced tools, can call one successfully, and the call is recorded as an mcp_call event for that run.

Advanced45 minutes

Create a safe plugin

Expose a focused Bun command without granting capabilities it does not use.

You will learnManifest modelProcess runtimeCapabilitiesCommands
  1. 1

    Create the plugin directory

    Add plugin.toml and index.ts. Keep each command focused on one operation.

  2. 2

    Declare the runtime

    Use a process command such as [\"bun\", \"run\", \"index.ts\"]. Add only the capability required by the command.

  3. 3

    Read newline JSON

    Read one request per line from stdin and return one JSON response per line. Send logs on separate non-JSON lines.

  4. 4

    Register a palette command

    Add a manifest command with a stable ID and a title that describes its result.

  5. 5

    Test denied capability behavior

    Remove the declared capability and confirm the host rejects the corresponding operation.

index.ts
for await (const line of console) {
  const request = JSON.parse(line);
  const response = { id: request.id, result: { ok: true } };
  console.log(JSON.stringify(response));
}
Checkpoint

The plugin loads, its command appears in the palette, valid requests return JSON, and undeclared host access fails.

Advanced35 minutes

Design a fan-out strategy

Use parallel agents where independent evidence is worth the execution cost.

You will learnSelection criteriaAgent diversityQueue limitsDecision records
  1. 1

    State the decision

    Write acceptance criteria and explicit comparison dimensions before launching runs.

  2. 2

    Select meaningful diversity

    Choose agents or prompts likely to produce independent approaches. Duplicate strategies add cost without information.

  3. 3

    Set a parallel limit

    Bound concurrent runs to the machine and provider limits. Remaining runs stay queued with durable state.

  4. 4

    Require the same verification

    Use one project configuration and the same detected checks for every candidate.

  5. 5

    Record the selection

    Create a Decision note that links the task, candidate runs, check outcomes, and selected pull request.

Cost rule

Fan out when the value of a better decision is higher than the cost of another independent run.

Advanced30 minutes

Automate with the CLI

Use the same core planning and execution logic outside the desktop workflow.

You will learnHeadless runsSearchWorktree controlComputer use
  1. 1

    Inspect worktrees

    Run asylum worktree list /path/to/repo and confirm managed branches resolve to absolute paths.

  2. 2

    Launch a headless run

    Use asylum run opencode \"Audit error handling\". Capture the process status in your shell automation.

  3. 3

    Search the result

    Use asylum search for a known symbol or phrase. Treat no matches as an empty result, not a command failure.

  4. 4

    Exercise computer-use commands

    Take a snapshot, then use click and fill only against deterministic coordinates and fields in a controlled environment.

  5. 5

    Add failure boundaries

    Stop automation when a run, worktree operation, or expected visual state fails. Preserve output for review.

Checkpoint

Your script should produce a durable run result and exit nonzero when a required step cannot complete.
